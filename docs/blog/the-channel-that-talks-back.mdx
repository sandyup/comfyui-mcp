_by [artokun](https://github.com/artokun) · June 16, 2026 · MCP · channels · case study_

[comfyui-mcp](https://github.com/artokun/comfyui-mcp) lets a Claude session
drive [ComfyUI](https://github.com/comfyanonymous/ComfyUI). This week it grew a
**[sidebar panel](/panel)** that runs on _your own_ Claude Code session — no LLM
API keys, no per-token billing. Building it meant pushing MCP somewhere it
doesn't normally go: **backwards**.

## MCP only flows one way

The Model Context Protocol is request/response with a clear direction. The
**host** (Claude Code, holding the model) calls **tools** on your **server**.
Your server answers. That's the whole loop, and it's the right shape for
"generate an image," "list my models," "queue this workflow" — the agent
decides, the tool does.

The panel breaks that assumption. Two things have to happen:

1. The agent edits the graph you're looking at — that's ordinary tool calls
   (`panel_add_node`, `panel_connect`, `panel_set_widget`). Easy: it's the
   normal direction.
2. **You type a message into the panel** — "wire the VAE into the decoder" —
   and it has to show up in the agent's conversation so Claude can act on it.
   That's the agent receiving input it didn't ask for. There's no tool call to
   hang it on. It's the _wrong direction_.

A server can't call the model. So how does a message that originates in a
browser sidebar end up in the prompt of a Claude Code session it has no handle
on?

## The pull fallback, and why it isn't enough

The boring-but-portable answer is a mailbox. The panel's message goes into a
queue; the server exposes a `panel_inbox` tool; the agent calls it to drain
whatever you typed. This works on **any** MCP host, and we keep it as the
fallback.

But polling is a bad conversation. The agent only "hears" you when it happens
to check the mailbox, so either it burns turns polling on a timer or it misses
you entirely while it's busy. What you want is for your message to just _appear_
in the conversation, the moment you send it, the same way a user turn does.

## The reverse channel

MCP has a second message type besides request/response: **notifications** —
one-way messages a server can emit without a reply. The host is free to ignore
them. But Claude Code understands one experimental kind: `claude/channel`. When
the host sees a `notifications/claude/channel` message, it injects the payload
into the running conversation as a `<channel>` block — exactly as if a new turn
arrived from outside.

That's the trick. The user types in the panel → the bridge hands the text to
the server → the server emits a notification → the host splices it into the
agent's prompt. No tool call, no polling, no API key. The flow runs backwards
through the same MCP connection:

```
panel ──ws──▶ comfyui-mcp ──notifications/claude/channel──▶ Claude Code ──▶ <channel> in the prompt
```

Two things have to be true for the host to accept it. You **declare the
capability** at construction:

```ts
capabilities: {
  tools: {},
  // Tell the host we speak this experimental channel. Without it,
  // every notification we send is silently dropped.
  experimental: { "claude/channel": {} },
}
```

…and you send the notification in the **exact shape** the host expects —
`{ content, meta }`, not a flat custom object:

```ts
server.server.notification({
  method: "notifications/claude/channel",
  params: {
    content: event.text,
    meta: { source: "comfyui-panel", kind: "user_message", tab_id, workflow },
  },
});
```

The `meta` is how the agent knows _which_ tab and workflow you spoke from —
the panel is multi-tab, so a message carries its origin and replies route back
to it.

## The bug that proved the shape matters

The first cut shipped the feature and it looked done — except panel messages
never reached the agent. No error. No log. The mailbox (`panel_inbox`) still
worked, so it _seemed_ like the channel just wasn't wired.

Two mistakes, both silent:

- **The capability wasn't declared.** A host that hasn't been told you speak
  `claude/channel` treats your notification as noise and drops it. Notifications
  have no reply, so there's nothing to error — the message just evaporates.
- **The params were a flat custom object**, not `{ content, meta }`. Even once
  the host was listening, it couldn't parse what we sent.

The fix ([v0.12.0](/changelog/2026-06-16)) was tiny — declare the capability,
reshape the params — but the lesson is the expensive part. **A one-way channel
fails silently by design.** Request/response tells you when you're wrong; a
dropped notification tells you nothing. So you can't lean on errors to find the
mismatch — you verify the contract by hand, on both ends, character for
character, because the protocol won't.

## Takeaways for MCP authors

1. **MCP can run backwards.** Tools are the request/response half; notifications
   are the server-initiated half. If your server needs to volunteer information
   the agent didn't ask for, that's a notification, not a tool.
2. **Capabilities are a handshake, not decoration.** A host only surfaces what
   you declared. Forget the declaration and your feature fails _silently_.
3. **Match the host's payload shape exactly.** `{ content, meta }` here — guessing
   a custom object gets you dropped, not an error.
4. **Keep a portable fallback.** Not every host speaks every experimental
   capability, so back the fancy push with a pull tool (`panel_inbox`) that works
   anywhere.
5. **Verify silent paths by observation, not by errors.** One-way messages don't
   complain. Watch the far end actually receive them.
